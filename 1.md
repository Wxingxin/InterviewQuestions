它们看起来相似，但**本质上是两个完全不同的机制**：

---

# ✔ **结论一句话**

**`line-height: 120%` 会被解析为相对“当前 font-size 的具体像素值”；
`line-height: 1.2` 是“unitless（无单位）值”，会被继承并作为 *比例* 继续影响子元素。**

**因此：`120%` 只继承计算后的像素值，`1.2` 继承比例。**

---

# ✔ 差异详细对比

| 写法                  | 类型                | 继承行为              | 子元素的 `font-size` 变化会影响行高吗？     | 常用吗？ |
| ------------------- | ----------------- | ----------------- | ------------------------------ | ---- |
| `line-height: 120%` | 相对值（有单位百分比）       | **继承的是计算后的 px 值** | ❌ **不会**（固定了 px 值）             | 不推荐  |
| `line-height: 1.2`  | **无单位（unitless）** | **继承比例**（1.2）     | ✔ **会**（子元素行高随 font-size 自动变化） | ✔ 推荐 |

---

# ✔ 示例说明（最直观差别）

假设父元素：

```css
.parent {
  font-size: 20px;
  line-height: 120%;  /* => 行高 = 20px * 1.2 = 24px */
}
```

子元素：

```css
.child {
  font-size: 12px;
}
```

### ❌ 情况1：`120%`

父的 `line-height` = **24px**（已经算死）
子继承的是 24px 行高，而不是 12px × 1.2：

* 子的字体 12px
* 行高仍然是父的 24px
  ➡ **行距过大，看起来很奇怪**

---

### ✔ 情况2：`1.2`

父行高 = 20px × 1.2 = 24px（在父里）
子继承的是“1.2 这个比例”，所以：

子行高 = 12px × 1.2 = **14.4px**

➡ **视觉更协调**

---

# ✔ 为什么业界推荐 `line-height: 1.2`（无单位）？

因为它是“字体倍数”概念，不固定像素：

* 保证文本在不同 `font-size` 下行距合理
* 避免子元素行距异常
* 更可控、响应式更好

像著名 CSS 规范：**Normalize.css / Bootstrap / Tailwind / 现代设计系统** 都使用无单位行高。

---

# ✔ 小总结（可背）

> `line-height: 120%` 继承的是具体像素，所以子元素字体变小时行距会过大；
> `line-height: 1.2` 继承比例（倍数），子元素行高会随 font-size 自动变化，因此更推荐。
